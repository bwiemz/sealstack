# CSL Codegen — Python `TypedDict` Record Types

**Date:** 2026-04-21
**Status:** Design, pending implementation
**Target milestone:** v0.3 (CSL codegen completeness)
**Scope:** `crates/sealstack-csl` codegen extension. No runtime changes, no new crates.
**Parent designs:**
- [`2026-04-21-csl-codegen-rust-wasm-design.md`](./2026-04-21-csl-codegen-rust-wasm-design.md) — Rust emit (spec-by-example for shape decisions).
- [`2026-04-21-csl-codegen-typescript-design.md`](./2026-04-21-csl-codegen-typescript-design.md) — TypeScript emit (closest analog; TypedDict is the direct Python equivalent of a TS `interface`).

---

## 0. Goal and non-goals

### Goal

Fill the `CompileTargets::PYTHON` stub so `sealstack compile` emits `out/py/generated.py` with `TypedDict` record types + `Literal` enum aliases + per-schema metadata constants for every schema in a CSL source. Python consumers — especially data-engineering teams working in Pandas / Polars / PyArrow / PySpark pipelines — get typed shapes that `json.loads()` output satisfies structurally, with zero runtime overhead and zero external dependencies.

### Non-goals (explicitly out of scope)

- **Pydantic v2 models.** The Python AI-agent ecosystem (LangChain, Instructor, OpenAI tool calling) expects Pydantic. That's a real adoption pull, but shipping Pydantic as the default binds generated code to Pydantic's release cadence (the v1→v2 migration was painful across the ecosystem) and forces a dependency on every Python consumer. A future `--python-flavor=pydantic` slice can emit sibling Pydantic models alongside the TypedDicts — not for this pass.
- **`@dataclass` output.** Nested dataclasses have a real foot-gun: `json.loads()` yields nested dicts, and `Customer(**data)` does not recursively lift nested dicts into nested dataclasses. Fixing that requires generated `from_dict` boilerplate per class. Out of scope; not the right shape for a plain-data pipeline either way.
- **Runtime validation.** Matches the Rust/TS "plain types, no validation library" discipline. Consumers who want validation can wrap with Pydantic (future slice) or construct their own `isinstance` checks.
- **Python version ceremony.** We target **Python 3.11+** as the minimum so we can use stdlib `typing.NotRequired` without a `typing_extensions` dependency. Python 3.11 is 2.5 years old (Oct 2022) and standard in modern data-eng and AI stacks.
- **SDK generation.** The `sdks/python/` crate is a separate deliverable.
- **Rich types for Ulid/Uuid/Instant.** Plain `str`. Consumers can brand locally if desired. Same discipline as Rust/TS.
- **Re-emitting the `sdks/python/` hand-rolled types.** Those are SDK surface DTOs, not schema record types. Orthogonal concern.

### Success criteria

- `cargo test -p sealstack-csl` gains a snapshot test. `--features slow-tests` gains a `mypy --strict` roundtrip test.
- `sealstack compile <project>` writes `out/py/generated.py` alongside the existing `out/{rust,ts}/generated.*`.
- The emitted file passes `mypy --strict` with zero errors and zero warnings.
- Byte-stable output: identical input → identical bytes.
- CSL schemas whose field names collide with Python keywords (`class`, `from`, `yield`, etc.) still emit valid Python via the TypedDict functional syntax. Compiler absorbs the target-language quirk so the user doesn't have to rename their fields.

## 2. Output file shape

Single file `out/py/generated.py`:

```python
# Generated by `sealstack compile`. Do not edit.
# ruff: noqa

from typing import Any, Final, Literal, NotRequired, TypedDict

Tier = Literal["free", "pro", "enterprise"]


class Customer(TypedDict):
    id: str
    external_id: str
    name: str
    domain: str
    tier: Tier
    owner: str
    health_score: NotRequired[float]
    summary: str
    created_at: str
    tenant: str


CUSTOMER_META: Final = {
    "NAMESPACE": "acme.crm",
    "SCHEMA": "Customer",
    "TABLE": "customer",
    "VERSION": 2,
    "PRIMARY_KEY": "id",
    "RELATIONS": {
        "tickets": {"target": "Ticket", "fk": "customer"},
        "contracts": {"target": "Contract", "fk": "customer"},
    },
}
```

### 2.1 Schema with keyword-colliding field name (functional syntax fallback)

```python
# Input CSL:
#   schema Holding {
#     id: Ulid @primary
#     class: String     // `class` is a Python keyword
#   }
#
# Emitted:

Holding = TypedDict(
    "Holding",
    {
        "id": str,
        "class": str,
        "tenant": str,
    },
)

HOLDING_META: Final = {
    "NAMESPACE": "default",
    "SCHEMA": "Holding",
    "TABLE": "holding",
    "VERSION": 1,
    "PRIMARY_KEY": "id",
    "RELATIONS": {},
}
```

The functional form accepts any string key, so reserved words survive to the wire. `NotRequired[T]` works inside the dict too (`"my_opt": NotRequired[int]`), so optionality survives as well. Consumers access fields via bracket notation: `holding["class"]`. This is idiomatic Python for dict-keyed data and happens to be what you already do for any TypedDict.

## 3. Primitive type mapping

| CSL             | Python                                 | Notes                                                                             |
| --------------- | -------------------------------------- | --------------------------------------------------------------------------------- |
| `String`        | `str`                                  |                                                                                   |
| `Text`          | `str`                                  |                                                                                   |
| `Ulid`          | `str`                                  | No brand.                                                                         |
| `Uuid`          | `str`                                  |                                                                                   |
| `Bool`          | `bool`                                 |                                                                                   |
| `I32`           | `int`                                  | Python `int` is arbitrary precision — no 2^53 footgun.                            |
| `I64`           | `int`                                  | Same. No JSDoc analog needed; Python ints just work.                              |
| `F32` / `F64`   | `float`                                |                                                                                   |
| `Instant`       | `str`                                  | RFC 3339 string. `datetime` is out of scope (rich-types, future slice).           |
| `Duration`      | `str`                                  |                                                                                   |
| `Json`          | `Any`                                  | Matches TS `unknown` at the "honest about any-shape" level. Forces explicit cast. |
| `Vector<N>`     | *skipped* (field omitted)             | Same as Rust/TS.                                                                  |
| `Ref<T>`        | `str`                                  | FK id.                                                                            |
| `List<T>`       | `list[T]`                              | Lowercase `list`, PEP 585 generic alias (3.9+).                                   |
| `Map<K, V>`     | *not emitted* (type checker rejects)   | `CslError::Codegen` if reached.                                                   |
| `Named(Enum)`   | the enum alias name (e.g., `Tier`)     |                                                                                   |
| `Named(Schema)` | `str`                                  | References to another schema render as the FK id.                                 |
| `T?`            | `NotRequired[T]` wrapping the inner     | See §5 for rationale over `T \| None`.                                            |

### 3.1 Why `int` for `I64` without a precision caveat

TypeScript needed a JSDoc warning because `JSON.parse` forces numeric JSON to `Number` (2^53-bounded). Python's stdlib `json` preserves arbitrary-precision integers — `json.loads('{"x": 9999999999999999999}')` returns a Python `int` with the full value. No precision loss. No JSDoc analog needed.

## 4. Enum emission

Python's idiomatic zero-runtime analog to TS's string literal union is `Literal[...]` aliased via `=`:

```csl
enum Tier {
  Free("free"),
  Pro,
  Enterprise("enterprise")
}
```

→

```python
Tier = Literal["free", "pro", "enterprise"]
```

**Rules:**

- Each variant's wire form is the string literal declared in parens, or the lowercased identifier if no paren-form is given (matches Rust / TS emit).
- Wire strings are escaped: `\` → `\\`, `"` → `\"`. Python string literals use double quotes for consistency with PEP 8 / ruff; single quotes would also work but PEP 8 picks one style per project and the one we're shipping is double.
- Enums are declared **before** any schema that references them. Ordering: `typed.enums` iterated in sorted key order (mirrors Rust / TS).
- Empty enum list: no `from typing import Literal` in the imports (see §9 for import gating).

### 4.1 Rejected alternatives

- **`enum.StrEnum`** (Python 3.11+) — requires `from enum import StrEnum`, instantiates a runtime class, consumers access via `Tier.Free` instead of the literal string. Breaks from the "plain data" discipline and adds runtime overhead for zero type-system gain.
- **`enum.Enum`** — same concerns as `StrEnum` plus .value dereferencing friction.
- **Python 3.12 `type` keyword** (`type Tier = Literal["free", "pro"]`) — requires 3.12+, which is too new. Stick with `=` alias for 3.11 compatibility.

## 5. Optional field syntax

`NotRequired[T]` inside a TypedDict field declaration:

```python
class Customer(TypedDict):
    id: str
    health_score: NotRequired[float]
```

**Why `NotRequired[T]` not `T | None`:**

- The Rust emitter uses `#[serde(default, skip_serializing_if = "Option::is_none")]`. `None` is **omitted** from the outbound JSON, and both "missing key" and "key with value `null`" deserialize to `None`. The Python shape that matches this exactly is `NotRequired[T]` — the key may be absent from the dict, and if present, carries a value of type `T`.
- `T | None` would require the key to always be present but allow a `None` value, at odds with `skip_serializing_if`'s omit-on-none behavior.
- `Optional[T]` (`= T | None`) — same issue as `T | None`. Misleading naming for the dict-missing-key semantics.

### 5.1 Interaction with `NotRequired` and the functional TypedDict form

```python
Holding = TypedDict(
    "Holding",
    {
        "id": str,
        "class": NotRequired[str],
    },
)
```

`NotRequired[T]` is a typing-level construct (not a class), and the functional TypedDict constructor accepts any typing expression as a value — including `NotRequired[T]`. Optional support survives the keyword-collision fallback path.

## 6. The `tenant` field

Same tightening as Rust and TS: `tenant: str` is required, no `NotRequired`. Every CSL-generated table carries a `tenant: text NOT NULL DEFAULT ''` column; the emitted Python interface requires the key to be present. Schemas that declare `tenant` with any type other than `String` / `Text` cause codegen to return `CslError::Codegen` — mirrors the existing guard in the Rust and TS emitters.

## 7. Metadata const

Per schema:

```python
CUSTOMER_META: Final = {
    "NAMESPACE": "acme.crm",
    "SCHEMA": "Customer",
    "TABLE": "customer",
    "VERSION": 2,
    "PRIMARY_KEY": "id",
    "RELATIONS": {
        "tickets": {"target": "Ticket", "fk": "customer"},
    },
}
```

- **Naming:** `<SCHEMA_NAME_SCREAMING_SNAKE>_META`. `Customer` → `CUSTOMER_META`, `AdminDoc` → `ADMIN_DOC_META`. PEP 8 convention for module-level constants. The conversion uses the same `to_snake` helper (via `super::to_snake`) and then `.to_uppercase()`.
- **`Final`** type-hint is a declaration that this name won't be rebound — type checkers narrow to the inferred literal shape. Consumers get `CUSTOMER_META["NAMESPACE"]` typed as `str` (technically `Literal["acme.crm"]` under sufficiently strict checkers), without `Final` it'd be broadened.
- **Relations:** `"tickets": {"target": "Ticket", "fk": "customer"}`. Relation key is a string (to sidestep Python-keyword issues on relation names: `"from": {...}` is legal dict, `from=...` isn't legal kwarg). Empty schemas get `"RELATIONS": {}` inline.
- **Cross-namespace references:** emit `"target": "hr.Employee"` verbatim. Same as Rust/TS.

Rejected alternative: a class with class attributes (`class CustomerMeta: NAMESPACE = "acme.crm"; ...`). Breaks from the "plain data" discipline; runtime class overhead; discourages dict-native bulk iteration used in pipelines.

## 8. Reserved-word collisions — functional TypedDict fallback

### 8.1 The problem

Python's class-syntax TypedDict requires field names to be valid Python identifiers. CSL's lowercase-start grammar allows field names that collide with Python keywords:

- Hard keywords (always reserved): `and`, `as`, `assert`, `async`, `await`, `break`, `class`, `continue`, `def`, `del`, `elif`, `else`, `except`, `finally`, `for`, `from`, `global`, `if`, `import`, `in`, `is`, `lambda`, `nonlocal`, `not`, `or`, `pass`, `raise`, `return`, `try`, `while`, `with`, `yield`
- Soft keywords (contextually reserved, 3.10+): `match`, `case`

A field named `class`, `from`, or `yield` would fail `class Foo(TypedDict): class: str` with a syntax error.

### 8.2 The fix

Per-schema detection: if any field in a schema is a Python keyword (hard or soft), emit via the **functional TypedDict syntax**:

```python
Holding = TypedDict(
    "Holding",
    {
        "id": str,
        "class": str,
        "tenant": str,
    },
)
```

The functional form accepts any string keys, so reserved words survive. `NotRequired[T]` wrapping still works inside the dict. `mypy --strict` accepts it as a fully-typed TypedDict, equivalent to the class form.

Schemas without any keyword-colliding field use the class form (cleaner, more idiomatic).

### 8.3 Keyword detection

```rust
fn is_python_keyword(name: &str) -> bool {
    matches!(
        name,
        "False" | "None" | "True"
        | "and" | "as" | "assert" | "async" | "await"
        | "break" | "class" | "continue"
        | "def" | "del"
        | "elif" | "else" | "except"
        | "finally" | "for" | "from"
        | "global" | "if" | "import" | "in" | "is"
        | "lambda" | "nonlocal" | "not" | "or"
        | "pass" | "raise" | "return" | "try"
        | "while" | "with" | "yield"
        // Soft keywords (3.10+) — treated as reserved for forward safety
        | "match" | "case"
    )
}
```

Per-schema choice: if `decl.fields.iter().any(|f| is_python_keyword(&f.name))`, emit functional form; otherwise class form. The `tenant` auto-injection never triggers this — `"tenant"` is not a Python keyword.

### 8.4 Rationale (cross-language symmetry)

Rust handles the same issue with raw identifiers (`r#class`). TypeScript sidesteps it because interface members aren't reserved-word-restricted in type position. Python's TypedDict has a functional form specifically for this case. The compiler absorbs each target's quirk so CSL's source of truth stays unified across targets.

## 9. File header and imports

```python
# Generated by `sealstack compile`. Do not edit.
# ruff: noqa

from typing import Any, Final, Literal, NotRequired, TypedDict
```

**Header:**

- Single `# Generated by ...` line matches the style of other targets.
- `# ruff: noqa` disables ruff linting on the file (analogous to TS's `/* eslint-disable */`). Narrower per-rule suppression is possible but brittle across ruff releases.
- No `# type: ignore` — we WANT mypy to validate the emitter's output. That's the point of the slow-tests roundtrip.

**Imports:**

- Always emit the full import line: `from typing import Any, Final, Literal, NotRequired, TypedDict`. Gating imports by feature presence (e.g., "no `Any` if no Json fields") adds code paths for negligible benefit; an unused import is a zero-cost no-op and ruff is disabled anyway.
- All imports are from `typing` (stdlib). No `typing_extensions`, no `enum`, no `dataclasses`. Zero-dep generated output.

## 12. Testing

### 12.1 Snapshot (insta)

`crates/sealstack-csl/tests/python_emit_snapshot.rs` — compiles the existing `rust_shapes.csl` fixture with `CompileTargets::PYTHON`, snapshots the emitted string. A second snapshot on `hello.csl`. A third fixture `python_keywords.csl` to exercise the functional-syntax fallback for reserved-word field names.

### 12.2 Unit tests

Per-helper tests inside `codegen/python.rs`:

- `render_field_type` on every primitive + `Optional` wrapping.
- `is_python_keyword` returns true for hard keywords, soft keywords (3.10+), and false for `tenant` / normal names / uppercase-start identifiers.
- `needs_functional_syntax(decl)` returns true if any field is a keyword, false otherwise.
- `escape_wire` handles `\` and `"`.
- `render_primitive` for each primitive.
- `Tenant` type guard: schema with `tenant: I32` returns `CslError::Codegen`.
- `Map<K,V>` returns `CslError::Codegen`.

### 12.3 `mypy --strict` roundtrip (slow-tests)

`crates/sealstack-csl/tests/python_emit_compiles.rs`, gated behind the existing `slow-tests` feature. Writes the emitted `generated.py` to a tempdir, populates a minimal `pyproject.toml` pinning `mypy>=1.11`, shells out to:

```bash
pip install --quiet mypy
python -m mypy --strict generated.py
```

Asserts exit 0 AND empty stderr. Skips gracefully with `skipping: python not on PATH` if `python --version` fails; analogous to the TS test's pnpm skip.

**Runtime toolchain:** prefers `uv` if present (faster), falls back to `python -m pip install mypy` if not. Both are mainstream modern Python dev setups.

### 12.4 CI integration

The `slow-tests` feature already runs in CI. Add `actions/setup-python@v5` (with Python 3.11) to the existing slow-tests CI job; mypy install takes ~5 seconds. If this job's wall-clock gets concerning, split the Python roundtrip behind its own feature (`python-mypy-roundtrip`) — for now, bundle.

## 13. Determinism

Same discipline as Rust and TS:

1. Enums in sorted key order, before schemas.
2. Schemas in `typed.decl_order`.
3. Fields within a schema in source order.
4. Relations within a meta `RELATIONS` dict in source order.
5. Namespace literal is `typed.namespace` verbatim; empty namespace renders as `"default"`.
6. Python keyword detection is a pure lookup, no ordering dependence.

Byte-stable output. Snapshot tests enforce this.

## 14. Risks and mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| mypy roundtrip flaky or slow (network, pip install, cache) | Medium | Separate `slow-tests` feature lets it be gated independently. Cache pip installs via CI cache action. Skip locally if Python unavailable. |
| `NotRequired` on a `list[T]` field breaks under older mypy | Low | mypy 1.11+ handles it. Pin via pyproject.toml in the test. |
| TypedDict functional syntax behaves differently for mypy `--strict` than class syntax | Low | mypy explicitly supports both forms with identical semantics. Test fixture with `python_keywords.csl` verifies. |
| Future Python adds a new soft keyword that collides | Low | The keyword list is a lookup. Adding a name requires one commit. Document in the spec. |
| Consumer's own file contains `from typing import TypedDict` with a custom definition | Low | Users don't typically shadow `typing.TypedDict`. Not our problem. |
| A CSL field named `tenant` but typed as `Json` | Low | The tenant guard catches this (requires String/Text). Same as Rust/TS. |

## 16. Summary

- One new module: `crates/sealstack-csl/src/codegen/python.rs`.
- Zero new crates, zero new deps on the Rust or Python side.
- CLI writes `out/py/generated.py` alongside the other targets.
- Three-tier testing: snapshot → unit → `mypy --strict` roundtrip (slow-tests).
- Mirrors the TS emit's discipline (plain types, zero runtime, no validation library) while respecting Python-specific realities: `NotRequired` for optional, `Literal` aliases for enums, `Final` dicts for metadata, and functional TypedDict syntax for keyword-colliding field names.
- Python 3.11+ minimum so stdlib `typing.NotRequired` suffices without `typing_extensions`.

After this lands, the CSL codegen matrix is complete for the three target languages that matter most to the current stack: Rust (servers, gateway, ingest), TypeScript (console, SDK), Python (data eng, AI pipelines).
