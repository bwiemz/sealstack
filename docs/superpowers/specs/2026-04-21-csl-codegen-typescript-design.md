# CSL Codegen — TypeScript Record Types

**Date:** 2026-04-21
**Status:** Design, pending implementation
**Target milestone:** v0.3 (CSL codegen completeness)
**Scope:** `crates/sealstack-csl` codegen extension. No runtime changes, no new crates.
**Parent design:** [`2026-04-21-csl-codegen-rust-wasm-design.md`](./2026-04-21-csl-codegen-rust-wasm-design.md) — Rust emit design, used as spec-by-example for shape decisions.

---

## 0. Goal and non-goals

### Goal

Fill the `CompileTargets::TYPESCRIPT` stub so `sealstack compile` emits `out/ts/generated.ts` with typed interfaces + enum types + schema-metadata constants for every schema in a CSL source. App developers and the in-tree SvelteKit console can then `import { Customer, CustomerMeta } from './generated'` and work with typed records instead of `Record<string, unknown>` / hand-rolled mirrors.

### Non-goals (explicitly out of scope)

- **Regenerating `console/src/lib/types.ts`.** Those are API-wire DTOs (SchemaMeta, SearchResponse, Receipt, etc.) — a different concern from per-schema record types. The console may eventually import the generated file for record typing; that's a follow-up.
- **Python emit.** Separate slice. The Rust + TypeScript designs together serve as spec-by-example for Python's Pydantic v2 conventions.
- **Runtime schema validation (zod, valibot, io-ts).** The console uses plain TypeScript + `svelte-check`, no validation-library deps. We match that discipline. Consumers who want runtime validation can wrap the generated types themselves.
- **Fetch client / SDK wrapper.** The `sdks/typescript/` crate is a separate deliverable.
- **Branded types for Ulid / Uuid / Instant.** Plain `string` by default. Consumers can brand in their own code.
- **`enum` keyword or const-enum output.** String literal unions only — see §4.
- **Multi-namespace disambiguation.** CSL v0.1's type checker enforces single-namespace per project (see `crates/sealstack-csl/src/types.rs`). If that invariant ever relaxes, add a flat-prefix collision pass then; out of scope now.
- **Runtime behavior: none.** Zero changes to the engine, gateway, or policy runtime.

### Success criteria

- `cargo test -p sealstack-csl` gains a snapshot test + `--features slow-tests` gains a `tsc --noEmit` roundtrip test. Both pass on all supported CSL fixtures.
- `sealstack compile <project>` writes `out/ts/generated.ts` alongside the existing `out/rust/generated.rs`.
- The emitted file compiles under `tsc --noEmit` with `strict: true`, no warnings.
- Byte-stable output: identical input → identical bytes. Same determinism discipline as Rust emit.

## 2. Output file shape

A single file `out/ts/generated.ts`:

```ts
// Generated by `sealstack compile`. Do not edit.
/* eslint-disable */
/* tslint:disable */

export type Tier = 'free' | 'pro' | 'enterprise';

export interface Customer {
  id: string;
  external_id: string;
  name: string;
  domain: string;
  tier: Tier;
  owner: string;
  /** CSL I64; values above 2^53 lose precision. */
  health_score?: number;
  summary: string;
  created_at: string;
  tenant: string;
}

export const CustomerMeta = {
  NAMESPACE: 'acme.crm',
  SCHEMA: 'Customer',
  TABLE: 'customer',
  VERSION: 2,
  PRIMARY_KEY: 'id',
  RELATIONS: {
    tickets: { target: 'Ticket', fk: 'customer' },
    contracts: { target: 'Contract', fk: 'customer' },
  },
} as const;
```

No module wrapper, no namespace. Top-level `export`s only. Enum types precede their referencing interfaces. Each schema emits two top-level symbols: the interface (`export interface <Name>`) and the metadata const (`export const <Name>Meta`).

`as const` on the metadata object is load-bearing: without it, `RELATIONS.tickets.target` would widen to `string`, losing the literal-type narrowing that makes the metadata useful for reflection.

## 3. Primitive type mapping

| CSL             | TS                                            | Notes                                                                    |
| --------------- | --------------------------------------------- | ------------------------------------------------------------------------ |
| `String`        | `string`                                      |                                                                          |
| `Text`          | `string`                                      |                                                                          |
| `Ulid`          | `string`                                      | No brand. Consumers can brand locally.                                   |
| `Uuid`          | `string`                                      | Same.                                                                    |
| `Bool`          | `boolean`                                     |                                                                          |
| `I32`           | `number`                                      | Safe: 2^31 ≪ 2^53.                                                       |
| `I64`           | `number`                                      | `/** CSL I64; values above 2^53 lose precision. */` JSDoc prefix.        |
| `F32` / `F64`   | `number`                                      |                                                                          |
| `Instant`       | `string`                                      | RFC 3339 string. No `Date`, no `Temporal` — matches Rust default.        |
| `Duration`      | `string`                                      |                                                                          |
| `Json`          | `unknown`                                     | `Record<string, unknown>` would lie — JSON values can be primitives.     |
| `Vector<N>`     | *skipped* (field omitted from the interface) | Matches Rust. Vectors live in the vector store, not on the record.       |
| `Ref<T>`        | `string`                                      | FK id. Same discipline as Rust.                                          |
| `List<T>`       | `T[]`                                         |                                                                          |
| `Map<K, V>`     | *not emitted* (type checker already rejects) | Belt-and-braces: emitter returns `CslError::Codegen` if reached.         |
| `Named(Enum)`   | the Rust enum type name                       | `tier: Tier`                                                             |
| `Named(Schema)` | `string`                                      | References to another schema render as the FK id.                        |
| `T?`            | `field?: T`                                   | Optional property syntax. See §5 for rationale over `\| null` / `\| undefined`. |

### 3.1 Why `number` for `I64` specifically

`JSON.parse()` produces a JavaScript `Number` for any JSON numeric literal. Typing the field as `string` would let `customer.external_id.startsWith('9')` typecheck, then throw `TypeError` at runtime because the runtime value is a number, not a string. Without a zod-style parse wrapper, the TS type **must** match the actual runtime shape. The 2^53 precision footgun is accepted explicitly via the JSDoc prefix; users with larger IDs should model them as `Ulid` (which stringifies without loss).

Any field whose type is `I64` gets the JSDoc tacked on — required and optional alike. The comment lives immediately above the field:

```ts
/** CSL I64; values above 2^53 lose precision. */
external_id?: number;
```

## 4. Enum emission

String literal union per the CSL enum:

```csl
enum Tier {
  Free("free"),
  Pro,
  Enterprise("enterprise")
}
```

→

```ts
export type Tier = 'free' | 'pro' | 'enterprise';
```

**Rules:**

- Each variant's wire form is the string literal declared in parens, or the lowercased identifier if no paren-form is given (matches Rust emit's behavior in §2.4 of the parent design).
- Wire strings are escaped the same way as the Rust emit: replace `\` → `\\`, `'` → `\'`. (Rust uses `"..."`; TS uses `'...'`. Same escape pair, different quote char.)
- Enums are declared **before** any schema that references them. Ordering: `typed.enums` iterated in sorted key order (mirrors the Rust emit's fix from PR #20's Phase A review).

### 4.1 Rejected alternatives

- **TypeScript `enum`** — generates an IIFE in emitted JS, has reverse-mapping footguns (`Tier[0]` vs `Tier.Free`), and the TypeScript team increasingly discourages it. Not worth the ceremony.
- **Const object + `typeof` type** (`const Tier = { Free: 'free', ... } as const; type Tier = typeof Tier[keyof typeof Tier]`) — useful when consumers need to iterate at runtime (`Object.values(Tier)`). Adds two exports per enum for a use case that's not load-bearing in the common case. A future slice can add sibling const objects alongside the types if demand shows up.

## 5. Optional field syntax

`field?: T` — the optional-property syntax. Used only when the CSL AST has `TypeExpr::Optional(inner)` at the outermost level.

**Why `?:` not `field: T | undefined` or `field: T | null`:**

- The Rust emitter uses `#[serde(default, skip_serializing_if = "Option::is_none")]`. That means `None` is **omitted** from the outbound JSON, and both "missing key" and "key with value `null`" deserialize to `None`. The TS shape that matches this exactly is `field?: T` — TypeScript treats missing property and `undefined`-valued property identically for structural compatibility.
- `field: T | null` would force consumers to write `customer.health_score === null` instead of `customer.health_score === undefined` — at odds with what the wire actually produces (`skip_serializing_if` omits, doesn't emit `null`).
- `field: T | undefined` is equivalent to `field?: T` at the type-system level but forces explicit property declaration on every consumer constructing a `Customer` literal. The `?:` form lets them omit missing fields.

### 5.1 Interaction with `exactOptionalPropertyTypes`

Consumers enabling TypeScript's `exactOptionalPropertyTypes` compiler option will see stricter behavior: `customer.health_score = undefined` requires the field to be declared optional (`?:`), not just `| undefined`. Our `?:` choice works cleanly under both settings.

## 6. The `tenant` field

Same tightening as the Rust emit (parent design §2.5): every CSL-generated table carries a `tenant: text NOT NULL DEFAULT ''` column, and the emitted TypeScript interface declares `tenant: string` **required** (no `?:`). A JSON payload missing the `tenant` field will fail to pass strict validation at the app's trust boundary (e.g., when `fetch`-parsed data is assigned to a `Customer`).

Schemas that explicitly declare a `tenant` field with any type other than `String` / `Text` cause codegen to return `CslError::Codegen`, identical to the Rust path. The type guard lives in the shared AST check, not duplicated.

## 7. Metadata const

Per schema:

```ts
export const CustomerMeta = {
  NAMESPACE: 'acme.crm',
  SCHEMA: 'Customer',
  TABLE: 'customer',
  VERSION: 2,
  PRIMARY_KEY: 'id',
  RELATIONS: {
    tickets: { target: 'Ticket', fk: 'customer' },
    contracts: { target: 'Contract', fk: 'customer' },
  },
} as const;
```

- **Naming:** `<SchemaName>Meta`. `Customer` → `CustomerMeta`. Not `customer_meta` (not idiomatic TS), not `Customer.META` (no class). A sibling const is the lightest export shape that still groups the metadata.
- **`as const`** is required: without it, `RELATIONS.tickets.target` widens to `string`, defeating the point of emitting literal types.
- **RELATIONS** is a plain object keyed by relation name. Each value is `{ target: string, fk: string }` (both narrow to literal types via `as const`). If a schema has no relations, emit `RELATIONS: {}` rather than omitting the field, so consumers can always reflect via `CustomerMeta.RELATIONS`.
- **Cross-namespace references:** if `rel.target` contains a `.` (qualified name), emit it verbatim: `target: 'hr.Employee'`. Matches Rust.

Rejected alternative: `static readonly` on a class. Classes with no methods are noise; an interface + a const pair is the TypeScript idiom.

## 8. File header

```ts
// Generated by `sealstack compile`. Do not edit.
/* eslint-disable */
/* tslint:disable */
```

- **No doc comment** describing namespace flattening — CSL enforces single-namespace at v0.1, so there's nothing to explain. If multi-namespace lands, add the comment then.
- **`/* eslint-disable */`** is broad but correct for a generated file. Narrower allows (e.g., `@typescript-eslint/no-explicit-any`) can't cover lints a consumer's config adds.
- **`/* tslint:disable */`** is belt-and-braces for legacy TSLint users — zero cost, harmless if unused.

Explicitly **not included**:

- `@ts-nocheck` — would kill all type-checking on the file itself; we want the emitter's output to be valid TypeScript, not tolerated TypeScript.
- `// @generated` marker — GitHub / some code-review tools collapse files with this marker, which is often desirable but the canonical form varies across ecosystems (`// @generated`, `// prettier-ignore`, `// @ts-nocheck`, etc.). Skip for v0.1; add if a consumer asks.

## 10. CLI changes

[`crates/sealstack-cli/src/commands/compile.rs::write_outputs`](../../../crates/sealstack-cli/src/commands/compile.rs) gains the TypeScript write-out, mirroring the Rust path:

```rust
let ts_dir = output_dir.join("ts");
std::fs::create_dir_all(&ts_dir)?;

if !out.typescript.is_empty()
    && !out.typescript.starts_with("// TypeScript codegen not yet implemented")
{
    std::fs::write(ts_dir.join("generated.ts"), &out.typescript)?;
}
```

The stub-check mirrors the Rust guard exactly — defensively avoids overwriting a vendored file while the target is still stubbed upstream. Once the stub is replaced by the real emitter (this slice), the check reduces to `!out.typescript.is_empty()` but keeping the `starts_with` guard costs nothing and leaves a clear intent signal.

## 11. Testing

### 11.1 Snapshot (insta)

`crates/sealstack-csl/tests/typescript_emit_snapshot.rs` — compile the existing `rust_shapes.csl` fixture with `CompileTargets::TYPESCRIPT`, snapshot the emitted string. Accepting the snapshot is a reviewable diff. Matches the Rust snapshot-test pattern exactly.

A second snapshot on a minimal fixture (`hello.csl`, already present) ensures single-schema files also emit correctly.

### 11.2 Unit tests

Per-type-mapper tests inside `codegen/typescript.rs`, analogous to the Rust emit's `type_mapper_tests` module:

- `render_field_type` on each primitive.
- Optional wraps the inner type with `?:`.
- Ref / Named(Schema) renders `string`.
- List renders `T[]`.
- Map returns `CslError::Codegen`.
- Enum wire-form escaping matches the Rust emit's escape pair (`\` → `\\`, `'` → `\'`).
- Tenant type guard: schema with `tenant: I32` returns `CslError::Codegen`.

### 11.3 `tsc --noEmit` roundtrip (slow-tests)

`crates/sealstack-csl/tests/typescript_emit_compiles.rs`, gated behind the existing `slow-tests` feature. Writes the emitted `generated.ts` to a tempdir, populates a minimal `package.json` pinning `typescript@5`, shells out to:

```bash
pnpm install --silent
pnpm exec tsc --noEmit --strict --exactOptionalPropertyTypes generated.ts
```

Asserts exit 0 and empty stderr. Takes ~15–30 s including `pnpm install`; caching via `pnpm` global store keeps subsequent runs fast.

**Fallback if pnpm isn't installed:** skip the test with a `println!("skipping: pnpm not on PATH")` rather than failing. The snapshot + unit tests catch most issues; the tsc roundtrip is a defense-in-depth check, not the primary correctness guarantee.

### 11.4 CI integration

The `slow-tests` feature already runs in CI (from PR #20's Phase D work). Add `Node.js` + `pnpm` setup to the existing `sealstack-csl --features slow-tests` CI job. If Node setup turns out to be a pain, gate the tsc roundtrip behind a *second* feature (`typescript-tsc-roundtrip`) that only CI enables — local developers can `cargo test --features slow-tests` and skip the tsc spin-up.

## 12. Determinism

Same discipline as Rust emit:

1. Enums in sorted key order, before any schemas that reference them.
2. Schemas in `typed.decl_order` (source order from the CSL file).
3. Fields within a schema in source order.
4. Relations within a `Meta.RELATIONS` object in source order.
5. Namespace literal is `typed.namespace` verbatim (for `NAMESPACE` values); empty namespace renders as `'default'`.

Identical input → identical bytes. Snapshot tests enforce this.

## 13. Risks and mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| tsc roundtrip is too slow or flaky (pnpm install, network) | Medium | Feature-gate the tsc check separately from the snapshot+unit tests. Snapshot covers the bulk of correctness. |
| Generated file fails under unusual consumer tsconfig (`noImplicitAny`, exotic moduleResolution) | Low | `tsc --strict` in the roundtrip covers the strictest realistic config. Consumers with weirder setups can disable lints on the generated file. |
| Enum wire forms with Unicode characters | Low | TS string literals handle Unicode natively. Escape only `\` and `'`. |
| Future `exactOptionalPropertyTypes` behavior change in TS | Low | Our `?:` form is stable across both settings. Documented in §5.1. |
| Console adopts the generated file and its existing hand-rolled types start to conflict | Low | Out of scope for this slice. When the console migrates, that's a separate PR. |

## 14. Open questions / deferred

1. **Per-namespace file splitting** — deferred until multi-namespace projects exist.
2. **Runtime validation helpers** — if a downstream slice adds zod / valibot support, we'd emit sibling schema objects (`export const CustomerSchema = z.object({ ... })`). Not for v0.1.
3. **Console migration** — swap `console/src/lib/types.ts`'s hand-rolled record mirrors for imports from `out/ts/generated.ts`. Natural follow-up once this lands.
4. **Fetch client generation** — typed wrappers around `/v1/query` that use the generated interfaces. Future SDK slice.
5. **`@generated` collapse marker** — can add later without invalidating any consumer code.

## 15. Summary

- One new module in `sealstack-csl/src/codegen/typescript.rs`.
- Zero new crates, zero new deps on the Rust side.
- CLI writes `out/ts/generated.ts` alongside existing outputs.
- Three-tier testing: snapshot → unit → tsc roundtrip (slow-tests).
- Mirrors the Rust emit's discipline (flat, no ceremony, no runtime deps) while respecting TS-specific realities (string literal unions over enums, `?:` over `| null`, `number` for I64 matching `JSON.parse`'s runtime reality).
- Deferred: Python (separate slice), validation, client generation, console migration.

After this lands, the only remaining CSL codegen targets are Python and whatever future languages users request. The Rust / TS pair is the load-bearing set for the current stack (Rust servers + TS consoles / SDK).
