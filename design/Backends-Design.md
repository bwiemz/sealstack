# Vector Store + Embedder Backends — Design Notes

**Companion to**: `crates/sealstack-common/`, `crates/sealstack-vectorstore/`, and `crates/sealstack-embedders/` in the drop-in.
**Status**: Unverified skeleton. ~1,843 lines across 11 files. Targets `qdrant-client` 1.12, `reqwest` 0.12, Rust 2024. Verify locally; the sandbox has no Rust toolchain.

---

## 1. Why these three crates together

`sealstack-engine` has been built against traits — `VectorStore`, `Embedder`, `PolicyEngine`, `Reranker`, `EngineHandle`. The engine doesn't care whether vectors live in a `DashMap` or a remote Qdrant cluster, or whether embeddings come from a hash function or Voyage's API. It only cares about the traits.

This drop-in adds the first real implementations of two of those traits:

| Crate | What it provides |
|---|---|
| `sealstack-common` | `SealStackError`, `SealStackResult`, identifier newtypes (`ContextId`, `TenantId`, `ChunkId`, …), `Tenant` record |
| `sealstack-vectorstore` | `VectorStore` trait, `Chunk` + `SearchResult` types, in-memory backend, Qdrant backend |
| `sealstack-embedders` | `Embedder` trait, stub embedder, Voyage embedder, OpenAI embedder |

`sealstack-common` is included because every other crate references its types; it's also dependency-light (no tokio, no sqlx, no network) so the workspace can build in tiers.

## 2. `sealstack-common` — the error and id foundation

### `SealStackError`

Seven variants — `NotFound`, `Unauthorized`, `Validation`, `Backend`, `Policy`, `Config`, `RateLimited`, plus a passthrough `Other`. Backends pick the right variant; the engine converts to `sealstack_engine::api::EngineError` at its boundary and the gateway converts to MCP `ToolError` at the wire.

A deliberate choice: **three error types across three layers**. Backend errors are concrete failure modes; engine errors frame the action semantics; gateway errors encode the wire protocol. Collapsing them into one would make the `Display` impls leak implementation details (or worse, cause us to paper over distinctions that matter, like "rate limited by Voyage" vs "policy denied by your tenant's rule").

### Identifier newtypes

`ContextId`, `SchemaId`, `TenantId`, `ConnectorId`, `ChunkId` — all newtypes over `ulid::Ulid`. Generated by a `macro_rules!` that also derives `Display`, `FromStr`, `Default`, and `Serialize`/`Deserialize` via a `#[serde(transparent)]` wrapper so the wire format is still a plain ULID string.

Upside: passing a `SchemaId` where a `ConnectorId` is expected is a compile error. Downside: you occasionally need `.0` to get at the inner `Ulid` for interop. Worth it.

## 3. `sealstack-vectorstore` — trait + two backends

### Trait surface

```rust
#[async_trait]
pub trait VectorStore: Send + Sync + 'static {
    fn kind(&self) -> &'static str;
    async fn ensure_collection(&self, name: &str, dims: usize) -> SealStackResult<()>;
    async fn ensure_collection_spec(&self, spec: &CollectionSpec) -> SealStackResult<()> { /* default */ }
    async fn upsert(&self, collection: &str, chunks: Vec<Chunk>) -> SealStackResult<()>;
    async fn search(&self, collection: &str, query_vec: Vec<f32>, top_k: usize, filter: Option<Value>) -> SealStackResult<Vec<SearchResult>>;
    async fn delete(&self, collection: &str, ids: Vec<ulid::Ulid>) -> SealStackResult<()>;
    async fn count(&self, collection: &str) -> SealStackResult<u64>;
    async fn drop_collection(&self, name: &str) -> SealStackResult<()>;
}
```

Two data types:

- `Chunk` — id, content, embedding vector, metadata map. Input to `upsert`.
- `SearchResult` — id, score, content, metadata. Output of `search`.

Note the dedicated `content` field on `SearchResult`. The earlier engine code read text from `metadata["content"]`; I unified on the explicit field and patched `sealstack-engine/src/retrieval.rs` accordingly — this turn fixed a silent inconsistency with ingestion that would have yielded empty excerpts on every real query.

### Memory backend

`InMemoryStore` wraps `DashMap<String, CollectionState>`, where `CollectionState` holds `dims`, `distance`, and `Vec<Chunk>`. Every `search` call walks the whole vector — linear cosine. Fine up to ~10k chunks; past that, switch to Qdrant.

Filters are applied post-search. The `matches_filter` helper accepts shallow equality only — strings, numbers, bools. Arrays and nested objects are silently rejected (treated as no-match). This matches what CSL `@facet` fields produce.

Cosine math is inline and obvious — `dot / (norm(a) * norm(b))`, clamped to `[-1, 1]`. Dot and Euclidean are also supported; Euclidean is returned as `-distance` so larger is better (keeping the sort direction consistent across metrics).

### Qdrant backend

The real production backend. Three interesting bits:

**Point-id encoding.** Qdrant accepts either UUIDs or unsigned 64-bit integers as point IDs. ULIDs are 128-bit Crockford-base32 strings. We encode each ULID as a UUID-shaped hex string via `uuid_from_ulid` / `ulid_from_uuid` — both are 128 bits, so the conversion is lossless. The dashes go in at the standard UUID positions (8-4-4-4-12) so Qdrant accepts the string as a valid UUID format.

**Value conversion.** Qdrant has its own `Value` type (`qdrant_client::qdrant::Value`) that's structurally similar to `serde_json::Value` but uses `Kind::StringValue`/`IntegerValue`/`DoubleValue`/`BoolValue`/`ListValue`/`StructValue` enum tags. We provide `serde_json_to_qvalue` and `qvalue_to_serde_json` that walk the tree recursively. Round-trip stable for all scalar types and arrays; objects nest correctly.

**Filter translation.** `value_to_filter` converts a narrow JSON DSL into Qdrant's `Filter`:

| JSON | Qdrant |
|---|---|
| `{ "status": "open" }` | `must: [FieldCondition(status matches "open")]` |
| `{ "count": 42 }` | `must: [FieldCondition(count matches 42)]` |
| `{ "tag": ["x", "y"] }` | `must: [FieldCondition(tag matches any of ["x", "y"])]` |

This covers every `@facet` filter CSL currently produces. More complex predicates (range, nested `should`/`must_not`, geo) need an explicit DSL upstream — when CSL gains an explicit query syntax, translate there rather than smuggling structure through the JSON shape.

### Feature flags

- `memory` (default) — enables `InMemoryStore`
- `qdrant` — enables `QdrantStore` and the `qdrant-client` dependency
- `all` — both

Gating keeps pure-stub users out of the qdrant-client compile closure, which is substantial (tonic + prost + rustls pulls in ~50 crates).

## 4. `sealstack-embedders` — trait + three backends

### Trait surface

```rust
#[async_trait]
pub trait Embedder: Send + Sync + 'static {
    fn name(&self) -> &str;
    fn dims(&self) -> usize;
    fn max_batch(&self) -> usize { usize::MAX }
    async fn embed(&self, texts: Vec<String>) -> SealStackResult<Vec<Vec<f32>>>;
}
```

`name()` is expected to match the CSL `context { embedder = "..." }` value. `dims()` must match the `vector_dims` in the schema — if they disagree, ingestion fails at the vector-store layer with a clear validation error (see `InMemoryStore::upsert`).

### Stub

BLAKE3 XOF over the input, expanded into `dims` f32s in `[-1, 1]`. Deterministic, cheap, semantically useless. Good enough for every test that doesn't measure retrieval quality — for example, the scaffolding brief's acceptance test just needs the pipeline to return a hit, not the right one.

### Voyage

HTTP client against `https://api.voyageai.com/v1/embeddings`. Default model `voyage-3` (1024 dims). A small catalog of `dims_for_model` handles the published models; unknown models fail at construction time with a clear error — better than a confusing 400 at first query.

Three configuration knobs on the builder: `with_endpoint` (proxy / mock), `with_input_type` (Voyage's `document` vs `query` hint — useful for asymmetric retrieval where query embeddings are computed differently from doc embeddings), `with_dims` (override for models not in the catalog).

Rate limiting: the client surfaces `429` as `SealStackError::RateLimited` — a distinct variant so the engine's retry logic can back off without confusing it with an actual failure.

### OpenAI

Same pattern, against `https://api.openai.com/v1/embeddings`. Default model `text-embedding-3-small` (1536 dims).

Two OpenAI-specific knobs:

- `encoding_format: "float"` — we always request JSON floats, never base64. Costs a ~30% response-size premium but parses trivially; base64 would save network bytes and require decoding.
- `with_dimensions(d)` — the v3 family supports output truncation server-side. Caller gets back a `d`-dimensional vector that the server has already re-normalized so cosine similarity is preserved. Use this to trade quality for storage cost.

Also compatible out-of-the-box with anything that speaks the OpenAI embeddings shape: Together, Groq, LiteLLM proxies, and `text-embeddings-inference` (the HuggingFace-maintained server). Point `with_endpoint` at them and it just works.

### Feature flags

- `stub` (default)
- `voyage` — enables the Voyage backend and `reqwest`
- `openai` — enables the OpenAI backend and `reqwest`
- `all` — all three

## 5. Engine integration

The gateway binary (`sealstack-gateway/src/bin/server.rs`) composes the stack:

```rust
let vector_store: Arc<dyn VectorStore> = if config.qdrant_url.is_empty() {
    Arc::new(sealstack_vectorstore::memory::InMemoryStore::default())
} else {
    Arc::new(sealstack_vectorstore::qdrant::QdrantStore::connect(&config.qdrant_url).await?)
};
let embedder: Arc<dyn Embedder> = Arc::new(sealstack_embedders::stub::StubEmbedder::new(64));
let engine = Engine::new_dev(engine_config, vector_store, embedder).await?;
```

For production, swap the stub embedder:

```rust
let embedder: Arc<dyn Embedder> = if let Ok(key) = std::env::var("VOYAGE_API_KEY") {
    Arc::new(sealstack_embedders::voyage::VoyageEmbedder::new(key)?)
} else if let Ok(key) = std::env::var("OPENAI_API_KEY") {
    Arc::new(sealstack_embedders::openai::OpenAIEmbedder::new(key)?)
} else {
    Arc::new(sealstack_embedders::stub::StubEmbedder::new(1024))
};
```

This env-based fallback is deliberately not in the binary yet — adding it requires enabling the `voyage` / `openai` features on the gateway's `sealstack-embedders` dep, which is a production-deployment decision (pulls in `reqwest`). Flip it on when you actually plan to embed.

### Dims must line up

The single most common misconfiguration: an embedder producing `N`-dim vectors against a schema with `vector_dims = M`. The vector store catches it on the first `upsert` with `SealStackError::Validation`; the ingestion pipeline surfaces the error with enough context to debug. Still, the right place to catch this earlier is at schema-apply time — `sealstack schema apply` should load the embedder and check `embedder.dims() == schema.context.vector_dims` before accepting the schema. That's a TODO for the CLI.

## 6. Known gaps — in priority order

1. **Voyage rerank client.** `sealstack-engine/src/rerank.rs` has an `HttpReranker` scaffold; a concrete `VoyageReranker` at `https://api.voyageai.com/v1/rerank` is a direct port of the embedder pattern. Maybe 80 lines.

2. **Qdrant filter richness.** Our `value_to_filter` handles equality and "any of"; range, nested boolean, geo, and payload-text match are all Qdrant features we don't expose. The blocker isn't implementation — it's the lack of a filter DSL upstream. Resolve at the CSL level with an explicit `filter { ... }` block, then translate.

3. **Qdrant HNSW tuning.** `ensure_collection_spec` uses Qdrant defaults. Real production deployments want to set `hnsw_config { m, ef_construct }` and `quantization_config` per collection. Thread through `CollectionSpec` additions.

4. **Dims-mismatch checks at schema-apply time.** Caught late right now — should fail at `sealstack schema apply` when the chosen embedder's `dims()` disagrees with the schema's `vector_dims`.

5. **OpenAI base64 encoding.** The vendor recommends it for bandwidth savings. We request `float`. Worth revisiting for high-throughput ingestion.

6. **Azure OpenAI support.** Uses a different auth header (`api-key` instead of `Authorization: Bearer`) and a deployment-in-path URL. A small `AzureOpenAIEmbedder` wrapping our `OpenAIEmbedder` with the header swap covers it.

7. **Local embedders.** `candle` (the Rust ML framework) can run `BGE-small-en-v1.5` locally with no GPU. A `CandleEmbedder` would let customers run fully air-gapped without giving up retrieval quality. ~300 lines + the model weights.

8. **Qdrant collection migration.** If a schema's `vector_dims` changes, the existing collection is wrong. Right now we log a validation error on the conflicting `ensure_collection`. A real migration path would version-suffix the collection name, re-ingest into the new one, atomically swap, and drop the old.

9. **Observability.** Every embedder call logs at `debug` with token count; every vector store call logs at `debug` with latency. Production wants `tracing`-integrated spans so the retrieval pipeline's embedding / vector / BM25 / rerank stages show up cleanly in Jaeger.

10. **Concurrent-safe Qdrant `ensure_collection`.** Two boot processes racing on the same collection could both see "doesn't exist" and both try to create. Qdrant itself handles this (the second create returns an error), but we don't currently swallow that error. Idempotency through an explicit existence check + retry on a specific error kind.

## 7. Verification checklist for first `cargo check`

Things most likely to need adjustment against the real compiler, in rough priority:

1. **`qdrant-client` API.** The crate went through multiple API refactors between 1.5 and 1.12. I used the builder forms (`SearchPointsBuilder::new(...)`, `CreateCollectionBuilder::new(...)`, `DeletePointsBuilder::new(...)`, `CountPointsBuilder::new(...)`, `UpsertPointsBuilder::new(...)`, `VectorParamsBuilder::new(...)`). If any of these don't exist or have different signatures in your specific 1.12.x, the fix is usually mechanical: check the current examples in the `qdrant-client` repo's `examples/` directory.

2. **`Qdrant::from_url().api_key().build()`.** That fluent pattern might be `Qdrant::from_url(url).with_api_key(key).build()` or `QdrantBuilder::from_url(url)...` in the actual 1.12. Expect one fix here.

3. **`PointId::from(uuid_string)`.** I construct `PointId { point_id_options: Some(PointIdOptions::Uuid(uuid)) }` explicitly. If there's a `From<String>` or `From<Uuid>` impl, simpler is better.

4. **`Condition::matches(field, value)`.** This is the 1.12 builder; `matches` may be split into `matches` / `matches_text` / `matches_any` depending on the value type. Check examples.

5. **`Filter::must(conditions)`.** Likely fine but double-check: 1.12 has `Filter::new(...)` with builder methods.

6. **`reqwest` default features.** The workspace sets `reqwest = { version = "0.12", default-features = false, features = ["rustls-tls", "json"] }`. My backends don't override, so they inherit these. If you see build failures about a missing TLS backend, that's the first thing to check.

7. **`blake3::Hasher::finalize_xof()`.** Returns `OutputReader`. The `fill` method on it writes into a slice. If the trait is called something different (e.g., `read_exact`), one mechanical fix.

8. **`insta`/`proptest` dev-dep features.** The `sealstack-common` and `sealstack-embedders` crates don't use them. Pure cosmetic if cargo complains.

The memory vector store, the stub embedder, and `sealstack-common` should all build on a clean Rust 1.83 toolchain without any external services. That's the smoke test I'd run first.
